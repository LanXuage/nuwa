# Nuwa 开发指南

## 引言

### 项目概述
Nuwa 是一个功能强大的 AI Agent 框架，支持 ReAct（推理与行动）模式、工具集成和对话管理等高级功能。其核心设计理念是“一切皆节点”，通过有向无环图（DAG）构建灵活的处理流水线，使开发者能够轻松组合和扩展 AI 能力。

### 文档目的
本指南旨在为 Nuwa 项目的开发者提供全面的开发原则和规范，涵盖架构设计、代码设计、代码开发、设计模型和代码管理等方面。通过遵循本指南，开发者可以确保代码的一致性、可维护性和可扩展性。所有约束、规范和要求均已编号索引，便于在代码注释中引用。

### 目标受众
- Nuwa 核心开发人员
- 贡献者
- 基于 Nuwa 构建应用的开发者

## 架构设计原则

### 整体架构
Nuwa 采用分层架构与有向无环图（DAG）处理流水线相结合的设计。核心哲学是：所有组件都是 `Node`，可以通过链式调用形成复杂的 AI Agent 工作流。

### 核心分层
1. **Base 层** (`src/nuwa/base.py`)
   - 定义抽象接口：`Node`、`MessagesManager`、`InputChunk`
   - 提供依赖管理和链式操作符（`>`）

2. **LLM 层** (`src/nuwa/llm.py`)
   - 实现与大语言模型（LLM）的基础交互
   - 支持流式和非流式响应

3. **Chat 层** (`src/nuwa/chat.py`)
   - 扩展 LLM 能力，增加对话管理
   - 集成消息存储和工具调用

4. **ReAct 层** (`src/nuwa/re_act.py`)
   - 实现 ReAct（推理-行动）模式
   - 支持多轮推理循环和结构化输出解析

5. **Tool 层** (`src/nuwa/tool.py`)
   - 管理外部工具集成
   - 提供装饰器注册和参数推断

6. **Storage 层** (`src/nuwa/qdrant.py`)
   - 提供消息持久化实现（基于 Qdrant 向量数据库）
   - 支持语义检索和分块存储

7. **Agent Group 层** (`src/nuwa/agent_group.py`)
   - 支持多智能体场景
   - 角色管理和黑名单机制

8. **工具模块** (`src/nuwa/search.py`, `src/nuwa/alarm.py`, `src/nuwa/vector.py`)
   - 提供搜索、闹钟、向量嵌入等实用工具

### 组件关系
- **继承层次**：`Node` → `OpenAI` → `ChatLLM` → `ReActAgent` → `GroupAgent`
- **组合关系**：`ChatLLM` 包含 `MessagesManager` 和 `ToolsManager`
- **数据流**：用户输入 → 系统提示格式化 → 历史检索 → LLM 调用 → 工具执行 → 输出生成

### 扩展点
- **自定义存储后端**：实现 `MessagesManager` 接口
- **自定义 LLM 提供商**：继承 `Node` 类
- **自定义工具处理**：通过 `hook_tool_call` 回调
- **自定义节点**：实现 `Node.run()` 方法

## 代码设计规范（编号索引：C-系列）

### 模块设计
- **C-001**：每个核心功能应位于独立的模块文件中。
- **C-002**：模块名应使用小写蛇形命名（如 `re_act.py`）。
- **C-003**：模块应提供清晰的导出列表（在 `__init__.py` 中定义）。

### 类设计
- **C-004**：使用抽象基类（ABC）定义接口。
- **C-005**：类名使用大驼峰命名法（如 `ReActAgent`）。
- **C-006**：遵循单一职责原则，每个类应有明确的目的。
- **C-007**：优先使用组合而非继承，除非存在明确的“is-a”关系。

### 接口设计
- **C-008**：公共 API 应保持稳定，避免频繁变更。
- **C-009**：使用类型提示明确参数和返回值类型。
- **C-010**：异步接口应使用 `async`/`await` 语法。
- **C-011**：异常设计：定义明确的异常层次，避免裸露的 `Exception`。

### 错误处理
- **C-012**：使用 `try-except` 捕获特定异常，避免捕获过于宽泛的异常。
- **C-013**：日志记录应包含足够上下文（使用 `logging` 模块）。
- **C-014**：用户可见的错误消息应友好且无技术细节。
- **C-015**：内部错误应记录详细堆栈信息。

### 日志记录
- **C-016**：使用标准的 `logging` 模块进行日志记录。
- **C-017**：日志级别：DEBUG（开发调试）、INFO（正常运行）、WARNING（潜在问题）、ERROR（错误）、CRITICAL（严重故障）。
- **C-018**：日志格式应包含时间戳、模块名、日志级别和消息。

## 代码开发规范（编号索引：D-系列）

### 编码风格
- **D-001**：遵循 PEP 8 规范。
- **D-002**：行长度限制为 88 个字符（Black 风格）。
- **D-003**：使用 4 个空格缩进，禁止使用制表符。
- **D-004**：导入顺序：标准库、第三方库、本地模块，每组内按字母排序。

### 命名约定
- **D-005**：变量名：小写蛇形命名（`session_id`）。
- **D-006**：函数名：小写蛇形命名（`get_messages`）。
- **D-007**：类名：大驼峰命名（`ReActAgent`）。
- **D-008**：常量：大写蛇形命名（`MAX_LOOP`）。
- **D-009**：私有成员：前缀下划线（`_private_method`）。

### 注释规范
- **D-010**：文档字符串使用 Google 风格（或 reStructuredText）。
- **D-011**：每个模块、类、公共方法应有文档字符串。
- **D-012**：复杂算法应有行内注释解释意图。
- **D-013**：TODO 注释应包含负责人和日期。

### 类型提示
- **D-014**：所有公共 API 必须使用类型提示。
- **D-015**：使用 `typing` 模块中的泛型（`List[str]`、`Optional[int]`）。
- **D-016**：复杂类型可定义 `TypedDict` 或 `Pydantic` 模型。

### 文档字符串示例
```python
async def get_messages(self, session_id: str, user_input: str = "") -> List[ChatCompletionMessageParam]:
    """
    检索与会话相关的历史消息。

    Args:
        session_id: 会话标识符
        user_input: 用户输入（用于语义检索）

    Returns:
        消息列表，格式符合 OpenAI ChatCompletion 接口

    Raises:
        ValueError: 当 session_id 无效时
        ConnectionError: 当存储后端不可用时
    """
```

## 设计模式应用（编号索引：P-系列）

Nuwa 框架中广泛使用了多种设计模式，以下为具体应用及其编号索引：

### 1. 抽象工厂模式
- **P-001**：`MessagesManager` 抽象基类定义了消息存储的接口，允许不同的实现（如 `QdrantMessagesManager`、内存存储等）无缝替换。

### 2. 策略模式
- **P-002**：`Node` 作为策略接口，不同 LLM 提供商（如 `OpenAI`、`Anthropic`）可以作为具体策略实现。

### 3. 装饰器模式
- **P-003**：工具注册使用装饰器 `@tools.tool()`，提供简洁的语法将普通函数转换为可调用工具。

### 4. 观察者模式
- **P-004**：`hook_tool_call` 回调机制允许外部观察者监控工具调用过程，实现自定义处理逻辑。

### 5. 模板方法模式
- **P-005**：`Node.run()` 定义了处理算法的骨架，子类通过实现具体步骤来定制行为。

### 6. 责任链模式
- **P-006**：节点通过 `__gt__` 方法（`>` 操作符）形成处理链，每个节点依赖前驱节点，构成责任链。

### 7. DAG（有向无环图）模式
- **P-007**：节点依赖系统创建 DAG 结构，支持复杂工作流中的分支与合并，避免循环依赖。

### 8. 单例模式
- **P-008**：`ToolsManager` 和 `AlarmManager` 通过模块级全局实例提供单例访问，确保工具和闹钟管理的全局一致性。

## 测试策略（编号索引：T-系列）

### 测试层级
- **T-001**：单元测试：测试单个函数或类的方法。
- **T-002**：集成测试：测试模块间的交互。
- **T-003**：端到端测试：测试完整工作流。

### 测试工具
- **T-004**：`pytest` 作为主要测试框架。
- **T-005**：`pytest-asyncio` 支持异步测试。
- **T-006**：`pytest-playwright` 用于浏览器自动化测试。
- **T-007**：覆盖率工具：`pytest-cov`。

### 测试规范
- **T-008**：测试文件命名：`test_*.py` 或 `*_test.py`。
- **T-009**：测试函数命名：`test_<功能>_<场景>_<预期>`。
- **T-010**：使用 fixture 共享测试资源。
- **T-011**：模拟外部依赖（API、数据库等）。

### 测试覆盖率
- **T-012**：目标覆盖率：核心模块 ≥ 90%。
- **T-013**：使用 `pytest --cov` 生成覆盖率报告。
- **T-014**：集成到 CI 中，覆盖率下降时失败。

### 示例测试
```python
@pytest.mark.asyncio
async def test_react_agent_with_tools(api_key: str):
    """测试 ReActAgent 与工具集成"""
    tools = ToolsManager()
    
    @tools.tool(name="get_weather", ...)
    async def get_weather(city: str):
        return {"desc": "晴朗"}
    
    agent = ReActAgent(model="gpt-4", ..., tools_manager=tools)
    async for chunk in agent.run({"user": "今天天气如何"}):
        assert chunk.state in ["DOING", "DONE", "END"]
```
